## 에스파크(SPARC) 프레임워크

<figure><img src="../../../.gitbook/assets/image (165).png" alt=""><figcaption></figcaption></figure>

**Specification, Pseudocode, Architecture, Refinement, Completion**의 약자

즉, “AI에게 바로 코딩을 시키지 말고, 요구사항 → 의사코드 → 아키텍처 → 반복 개선 → 완료 검증” 순서로 개발을 진행하게 하는 방법론

<table><thead><tr><th width="164.4453125">단계</th><th width="107.63671875">의미</th><th>하는 일</th></tr></thead><tbody><tr><td><strong>S: Specification</strong></td><td>명세</td><td>요구사항, 목표, 범위, 제약조건, 성공 기준 정의</td></tr><tr><td><strong>P: Pseudocode</strong></td><td>의사코드</td><td>실제 코드 전에 로직 흐름을 자연어·알고리즘 형태로 정리</td></tr><tr><td><strong>A: Architecture</strong></td><td>아키텍처</td><td>파일 구조, 모듈, 데이터 흐름, API, DB 구조 설계</td></tr><tr><td><strong>R: Refinement</strong></td><td>개선</td><td>구현, 테스트, 디버깅, 리팩토링을 반복</td></tr><tr><td><strong>C: Completion</strong></td><td>완료</td><td>검증, 문서화, 배포 준비, 최종 리뷰</td></tr></tbody></table>

### 세션이 끊기지 않나...?

> **세션 끊김 문제 — 실제로 있음**
>
> Claude Code 맥락에서 컨텍스트 압축 자동 작동함:
>
> * 대화 길어지면 이전 메시지 자동 요약
> * 요약본 + 남은 미압축 컨텍스트로 계속 진행
> * 작업 자체는 끊기지 않음, 단 디테일 손실 가능
>
> 실제 위험 요소:
>
> 1. 컨텍스트 압축 중 세부 요구사항 유실
> 2. 모델이 초기 Specification 잊고 표류
> 3. 오래된 결정 재논의 반복
>
> \[대응 전략]
>
> 1. SPARC 각 단계 문서화 필수 CLAUDE.md 또는 .claude/sparc-spec.md 에 S/P/A 단계 결과물 저장 → 새 세션도 읽고 재개 가능
> 2. TaskCreate 활용 단계별 Task 생성 → 진행 상태 persistent하게 추적
> 3. 단계별 커밋 각 단계 완료 시 git commit → 코드 자체가 체크포인트
> 4. 큰 범위 → 분할 위임 /sparc 전체 → 너무 큼 /sparc S단계만 → 명확한 산출물 → 다음 세션에 P단계
>
> \[핵심]
>
> 세션 자체는 안 끊김. 컨텍스트 압축으로 이어짐. 위험은 디테일 유실. 방어책: 각 SPARC 단계 결과를 파일로 저장해두면 세션 재시작해도 CLAUDE.md 읽고 재개 가능.



## 바이브 코딩과의 차이점

> **핵심 차이:**\
> \*\*바이브 코딩은 “AI와 대화하며 감으로 빠르게 만드는 방식”\*\*이고, \*\*에이전틱 엔지니어링은 “AI 에이전트를 통제 가능한 개발 프로세스 안에서 활용하는 방식”\*\*입니다.
>
>
>
> ### 1. 바이브 코딩이란
>
> 바이브 코딩은 개발자가 자연어로 “이런 느낌의 앱 만들어줘”, “이 기능 추가해줘”, “에러 고쳐줘”처럼 지시하고, AI가 생성한 코드를 빠르게 받아보며 수정해 나가는 방식입니다. 주로 **아이디어 검증, 프로토타입, MVP, 개인 프로젝트**에 잘 맞습니다.
>
> 예를 들면:
>
> > “로그인 화면 만들고, 토스 앱처럼 깔끔한 느낌으로 해줘.”\
> > “이 버튼 누르면 결제 페이지로 이동하게 해줘.”\
> > “에러 나는데 고쳐줘.”
>
> 이 방식은 속도가 빠르고 진입 장벽이 낮지만, 코드 구조·보안·테스트·유지보수성이 약해질 수 있습니다. 최근 연구에서도 바이브 코딩은 인간이 대화형 프롬프트로 실험하고 탐색하는 방식에 가깝다고 설명됩니다. ([arXiv](https://arxiv.org/html/2505.19443v1?utm_source=chatgpt.com))
>
>
>
> ### 2. 에이전틱 엔지니어링이란
>
> 에이전틱 엔지니어링은 AI를 단순 코드 생성기가 아니라 **목표를 이해하고, 계획을 세우고, 파일을 수정하고, 테스트를 실행하고, 결과를 검증하는 에이전트**로 활용하는 방식입니다. 다만 핵심은 AI가 마음대로 만들게 두는 것이 아니라, 사람이 **요구사항, 제약조건, 아키텍처, 테스트 기준, 리뷰 절차**를 설계한다는 점입니다.
>
> 예를 들면:
>
> > “이 이슈를 해결하기 위해 관련 파일을 분석하고, 수정 계획을 세운 뒤, 테스트를 추가하고, PR 형태로 변경사항을 정리해줘.”\
> > “기존 인증 모듈 구조를 유지하면서 OAuth 로그인 기능을 추가해. 테스트 커버리지는 유지하고, 보안상 토큰 저장 방식은 기존 정책을 따라.”
>
> 즉, 개발자는 “프롬프트를 잘 던지는 사람”이 아니라 **AI 에이전트를 지휘하는 엔지니어링 리드**에 가까워집니다. IBM도 agentic engineering을 AI가 전체 코드베이스를 무작정 만드는 것이 아니라, agentic programming을 엔지니어링 도구로 활용하는 접근으로 설명합니다. ([IBM](https://www.ibm.com/think/topics/agentic-engineering?utm_source=chatgpt.com))
>
>
>
> ### 3. 비교하면
>
> <img src="../../../.gitbook/assets/image (160).png" alt="" data-size="original">
>
>
>
> ### 4. 가장 쉬운 비유
>
> **바이브 코딩**은 디자이너에게 “이런 느낌으로 포스터 하나 만들어줘”라고 말하며 즉석에서 결과물을 고치는 것에 가깝습니다.
>
> **에이전틱 엔지니어링**은 프로젝트 매니저가 작업지시서, 품질 기준, 검수 절차를 정해두고 여러 작업자를 움직여 실제 납품 가능한 결과물을 만드는 것에 가깝습니다.
>
>
>
> ### 5. 실무적으로는 이렇게 나뉩니다
>
> **바이브 코딩이 좋은 경우**는 아이디어를 빠르게 눈으로 확인해야 할 때입니다. 예를 들어 랜딩 페이지, 관리자 화면 초안, 내부 데모, PoC, 간단한 자동화 스크립트가 해당됩니다.
>
> **에이전틱 엔지니어링이 필요한 경우**는 결과물이 오래 운영되거나, 고객 데이터·결제·보안·권한·기존 시스템과 연결되는 경우입니다. 이때는 AI가 코드를 잘 짜는지보다 **요구사항이 명확한지, 테스트가 있는지, 롤백 가능한지, 사람이 리뷰했는지**가 더 중요합니다.
>
>
>
> ### 한 줄 요약
>
> **바이브 코딩은 “AI로 빨리 만들어보는 개발 방식”이고, 에이전틱 엔지니어링은 “AI 에이전트를 엔지니어링 프로세스 안에 넣어 통제·검증하며 개발하는 방식”입니다.** 실제 업무에서는 바이브 코딩으로 빠르게 방향을 잡고, 에이전틱 엔지니어링으로 제품화하는 흐름이 가장 현실적입니다.



## 하네스와의 관계

### 하네스(Harness)란?

{% hint style="info" %}
Agent = Model + Harness

업계의 유명한 구절. 에이전트는 모델과 하네스의 합.
{% endhint %}

모델 주변을 감싸고 제어하는 모든 코드, 인프라, 규칙, 피드백 시스템 총체.

<table><thead><tr><th width="263.16796875">하네스의 주요 구성 요소</th><th>역할 설명</th></tr></thead><tbody><tr><td>도구 및 인프라 (Tools &#x26; Sandbox)</td><td>에이전트가 안전하게 코드를 실행할 수 있는 격리된 환경(Sandbox), 파일 시스템, 외부 API(MCP 서버 등) 제공</td></tr><tr><td>결정적 제어 루프 (Control Loops)</td><td>에이전트가 실행 후 스스로 검증(Self-Verification)하고 린터(Linter)나 테스트 코드를 돌려 에러가 나면 스스로 고치게 만드는 피드백 파이프라인</td></tr><tr><td>컨텍스트/메모리 관리</td><td>너무 많은 토큰이 낭비되지 않도록 작업에 꼭 필요한 기억만 선별하여 공급하고 장기 기억 표준(예: <code>AGENTS.md</code>) 관리</td></tr><tr><td>안전장치 (Guardrails)</td><td>악의적인 프롬프트 주입(Injection)을 차단하거나 에이전트의 비정상적인 무한 루프, 탈옥 방지</td></tr></tbody></table>

### 에이전틱 엔지니어링과 하네스의 관계

이 둘은 '학문/패러다임(과정)'과 '구체적인 구현체(도구)'의 관계를 가집니다.

* 포함 관계: 에이전틱 엔지니어링이라는 거대한 목표를 달성하기 위해, 개발자가 가장 고도화하고 집중해야 하는 핵심 엔지니어링 대상이 바로 하네스 설계(Harness Engineering)입니다.
* 역할의 변화: 에이전틱 엔지니어링 시대의 개발자는 코드를 짜는 사람이 아니라 "하네스를 설계하고 진화시키는 사람"이 됩니다. 에이전트가 반복적으로 실수를 범한다면, 프롬프트를 고치는 게 아니라 하네스에 규칙(Assertion)을 추가하거나 검증 파이프라인을 개선하여 시스템적으로 실수를 차단합니다.
* 모델과의 독립성: 좋은 하네스는 특정 AI 모델에 종속되지 않습니다. 에이전틱 엔지니어링을 잘 해두면 내부 모델을 OpenAI에서 Anthropic이나 Gemini로 바꾸더라도 외부를 감싸고 있는 하네스(보안, 검증 루프, 도구)가 견고하기 때문에 시스템이 안정적으로 유지됩니다.

결론적으로, 에이전틱 엔지니어링은 AI 에이전트 기반의 개발 방법론 전체를 뜻하고, 하네스는 그 에이전트가 탈선하지 않고 프로덕션 환경에서 완벽하게 구동되도록 지탱하는 핵심 제어 아키처라고 이해하시면 됩니다.

> Rules
>
> 에이전트 행동 제약/지침. 항상 적용되는 정적 컨텍스트.
>
> * CLAUDE.md, system prompt에 삽입
> * "테스트 없이 코드 수정 금지", "한국어로 응답" 같은 규칙
> * 에이전트가 매 응답마다 참조
>
>
>
> Skills
>
> 재사용 가능한 절차적 지식 묶음. 특정 작업 수행법을 캡슐화.
>
> * /commit, /code-review 같은 슬래시 커맨드로 호출
> * 프롬프트 + 도구 조합을 템플릿화
> * 에이전트가 "어떻게 할지" 모를 때 참조하는 레시피
>
>
>
> MCP Servers
>
> Model Context Protocol — 외부 시스템 연결 표준화 레이어.
>
> * DB, API, 파일시스템 등을 도구(tool)로 노출
> * 에이전트 ↔ 외부 세계 인터페이스
> * 예: GitHub MCP, Slack MCP, Google Drive MCP
>
>
>
> LSP
>
> Language Server Protocol — 코드 인텔리전스 제공.
>
> * 정의 이동(go-to-definition), 타입 정보, 참조 찾기
> * 에이전트가 코드 수정 전 심볼 위치 정확히 파악
> * IDE와 에이전트 사이 코드 이해 공유
>
>
>
> Hooks
>
> 이벤트 기반 자동 실행. 특정 트리거에 셸 커맨드 실행.
>
> * PreToolUse, PostToolUse, SessionStart 등
> * 예: 파일 저장 시 자동 린트, 커밋 전 테스트 실행
> * 에이전트 행동에 부수효과(side effect) 삽입
>
>
>
> Sub-agents
>
> 독립 실행 에이전트. 메인 에이전트가 태스크 위임.
>
> * 병렬 실행으로 속도 향상
> * 메인 컨텍스트 오염 방지 (결과만 요약 반환)
> * 전문화된 역할 분리: investigator/builder/reviewer
>
> ***
>
> 관계 요약:
>
> Rules → "무엇을 하지 말 것"
>
> Skills → "어떻게 할 것"
>
> MCP Servers → "무엇에 접근할 것"
>
> LSP → "코드를 어떻게 이해할 것"
>
> Hooks → "언제 자동 실행할 것"
>
> Sub-agents → "누가 나눠서 할 것"
